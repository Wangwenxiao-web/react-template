# 后台管理系统模板改造方案

## 1. 文档目标

本方案用于将当前项目从“前端演示模板”改造成“可复用的后台管理系统模板”。

约束如下：

- 不走兼容性补丁路线，直接按后台模板目标重构关键骨架
- 不做过度设计，只保留后台模板必要能力
- 每一阶段都要求能形成稳定交付，而不是堆积半成品
- 后续实现必须沿着统一架构推进，避免页面先行、基础设施滞后

---

## 2. 当前现状判断

当前项目已经具备一部分工程底座，但距离后台模板还有明显差距。

已有基础：

- `React 19 + TypeScript + Vite` 已可正常开发、构建
- `ESLint + Prettier + Stylelint + Husky + Commitlint` 已接入
- 已有基础 UI 组件、主题变量、`antd` 颜色映射
- 已有登录页和路由懒加载的初步结构

当前问题：

- 首页仍是颜色系统演示，不是后台主框架
- 路由没有后台布局层、权限层、异常页层
- 登录逻辑仍是硬编码 mock，未形成真实认证链路
- 请求层只有最薄封装，缺少错误模型、鉴权失效处理、统一响应契约
- `api`、`stores`、`types`、`constants` 仍是占位目录
- 登录页未纳入 ESLint 约束，说明代码质量边界不一致
- UI 栈存在 `shadcn` 与 `antd` 并存，但页面级规范尚未统一

结论：

当前项目适合作为“后台模板改造起点”，但不适合作为“后台模板成品”直接复用。

---

## 3. 总体目标

目标不是做一个具体业务系统，而是做一个可以复用的后台模板，至少应满足：

- 有稳定的后台应用骨架
- 有完整的登录、会话、权限链路
- 有标准化的数据请求与页面开发范式
- 有清晰的目录边界与工程约束
- 有可扩展的主题系统与组件层
- 有基础测试与交付门禁

---

## 4. 优先级总览

### P0：先完成后台模板最小骨架

这是项目能否继续健康推进的分水岭，必须最先完成。

包含：

- 后台布局骨架
- 登录与会话基础链路
- 路由分层与页面分层
- 请求层标准化
- 页面级 UI 规范定稿

### P1：补齐后台模板高频开发能力

这是让模板真正“可复用”的阶段。

包含：

- 菜单、面包屑、标签页
- 列表页模板
- 表单页模板
- 权限控制
- 异常页与通用空态

### P2：补齐工程化闭环

这是让模板长期可维护的阶段。

包含：

- 测试
- mock
- 环境配置
- 打包与性能治理
- 文档与开发约束

### P3：增强模板产品力

这是在基础稳定后再做的增强项，不应前置。

包含：

- 国际化
- 主题切换
- 可配置化 dashboard
- 更丰富的高级组件沉淀

---

## 5. 分阶段实施方案

## P0：后台模板最小骨架

### 目标

让项目从“演示工程”变成“可承载后台业务的基础模板”。

### 必做项

#### 5.0.1 路由重构

将当前路由重构为至少三层：

- 公共路由层：`/login`
- 后台受保护路由层：`/`
- 异常路由层：`/403`、`/404`

要求：

- 后台区必须使用统一 `AdminLayout`
- 路由元信息必须标准化，至少包含 `title`、`icon`、`requiresAuth`、`hiddenInMenu`
- 禁止继续把页面直接挂在根 `App.tsx` 上做切换展示

#### 5.0.2 应用骨架

新增后台主框架：

- 侧边栏
- 顶栏
- 面包屑区域
- 主内容容器
- 全局滚动与内容宽度规范

要求：

- 布局结构先稳定，再继续开发页面
- 所有后台页面都必须挂在统一壳子内
- 页面内容区必须有统一间距和容器规范

#### 5.0.3 登录与会话

登录页必须从“动画 demo”转成“后台登录模板”。

要求：

- 登录表单回归后台模板本质，保留适度视觉设计，但不能继续以动画交互为核心
- 认证状态必须统一管理
- 登录成功后拉取当前用户信息
- 未登录访问后台页时自动跳转到登录页
- 登出必须清理会话并跳转登录页

#### 5.0.4 请求层标准化

请求层必须从 axios 包装升级为后台模板可用的统一能力。

必须补齐：

- 统一 `baseURL`
- token 注入
- 401 失效处理
- 业务错误码处理
- 统一错误提示入口
- 分页请求与分页响应类型
- 文件下载能力

要求：

- 不允许页面自己处理一套错误模型
- 不允许页面到处手写 token 逻辑

#### 5.0.5 数据层边界

必须明确状态边界：

- 服务端数据：查询库管理
- 客户端状态：全局 store 管理

建议：

- 服务端状态使用 `TanStack Query`
- 客户端状态只存放会话、布局、主题、标签页等 UI 状态

#### 5.0.6 UI 技术路线定稿

当前同时存在 `shadcn` 和 `antd`，后台模板必须尽快定一条主线。

建议：

- 页面级组件统一以 `antd` 为主
- 现有 token 系统继续作为主题源
- `shadcn` 仅保留为少量底层原子组件补充，不作为页面主栈

原因：

- 后台系统高频依赖表格、表单、弹层、选择器、日期组件
- `antd` 在后台模板场景下更适合作为主组件体系

### 交付物

- 可用的 `AdminLayout`
- 可用的登录、登出、路由守卫
- 标准化路由定义
- 统一请求层
- 统一会话管理

### 验收标准

- 未登录时无法进入后台路由
- 登录后可以稳定进入后台首页
- 刷新页面后可恢复会话或正确跳登录
- 后台页面已运行在统一布局下
- 页面不再依赖 `App.tsx` 做演示切换

---

## P1：后台高频页面范式

### 目标

让模板具备真实后台项目的日常开发承载能力。

### 必做项

#### 5.1.1 菜单系统

实现菜单配置与路由配置联动。

要求：

- 菜单由路由元信息驱动
- 支持一级菜单与二级菜单
- 支持隐藏菜单项
- 支持当前路由高亮
- 支持折叠侧边栏

#### 5.1.2 面包屑与页面标题

要求：

- 面包屑由路由链路自动生成
- 页面标题与路由元信息统一
- 页面头部区域应有统一组件承载

#### 5.1.3 标签页导航

后台模板建议内建标签页导航。

要求：

- 打开页面后可加入标签页
- 支持关闭当前、关闭其他、关闭全部
- 标签页状态由全局 store 管理

#### 5.1.4 Dashboard 首页

新增一个标准后台首页，不再使用颜色系统演示页作为主页。

首页建议包含：

- 欢迎区
- 数据概览卡片
- 最近操作或通知区域
- 图表或占位分析区

要求：

- 先做通用模板，不嵌入业务字段
- 结构上可复用，不做一次性视觉页面

#### 5.1.5 列表页模板

后台模板必须沉淀列表页模式。

必须具备：

- 查询表单
- 表格区
- 分页器
- 行操作
- 批量操作
- loading、空态、错误态

应抽象的通用能力：

- 查询参数与分页参数同步
- 刷新与重置
- 列显隐基础能力
- 状态标签展示

#### 5.1.6 表单页模板

后台模板必须沉淀新增/编辑页模式。

必须具备：

- 表单分区
- 校验
- 提交中状态
- 提交成功后的跳转或回填
- 返回前脏状态提示

#### 5.1.7 异常页

至少实现：

- `403`
- `404`
- `500`

要求：

- 风格统一
- 路由接入完整
- 可直接作为模板页面复用

### 交付物

- 菜单系统
- 面包屑系统
- 标签页系统
- Dashboard 页面
- 列表页模板
- 表单页模板
- 异常页

### 验收标准

- 新增一个后台页面时，只需补路由和页面组件即可接入菜单与面包屑
- 新建一个列表页时，不需要从零搭表格交互
- 新建一个表单页时，不需要从零搭提交流程

---

## P2：工程化闭环

### 目标

让模板从“能跑”进入“能长期维护和协作”。

### 必做项

#### 5.2.1 目录结构重构

当前 `views` 聚合方式不适合后台模板持续扩展。

建议采用分层结构，例如：

- `src/app`
- `src/pages`
- `src/widgets`
- `src/features`
- `src/entities`
- `src/shared`

最低要求：

- 路由、布局、认证、接口、通用组件、业务页面必须分层
- 不允许继续把所有页面堆在 `views`

#### 5.2.2 类型体系

必须补齐：

- 通用响应模型
- 分页模型
- 用户与权限模型
- 表单 DTO
- 路由元信息类型

要求：

- 类型定义不允许继续停留在占位状态
- 页面内临时类型只能作为局部实现细节，公共模型必须集中管理

#### 5.2.3 环境配置

至少补齐：

- `.env.development`
- `.env.test`
- `.env.production`
- `env.d.ts` 或等价环境变量类型定义

要求：

- API 地址、应用标题、构建标识等配置必须环境化
- 禁止把环境差异写死在代码中

#### 5.2.4 质量门禁修复

必须处理：

- 移除对 `src/views/Login.tsx` 的 ESLint 忽略
- 修正 `components.json` 指向不存在样式文件的问题
- 补齐必要 lint 规则或目录约束

要求：

- 不能保留“局部页面不受约束”的例外

#### 5.2.5 测试

后台模板不需要一开始就高覆盖率，但必须有最小闭环。

建议优先补：

- 路由守卫测试
- 请求层测试
- 登录流程测试
- 一个列表页流程测试

#### 5.2.6 Mock 机制

建议接入本地 mock。

要求：

- 开发阶段可脱离真实后端调试基础流程
- 登录、用户信息、列表、详情至少可 mock

#### 5.2.7 构建与性能

必须关注：

- 路由级代码分割
- 页面首屏 chunk 控制
- 依赖体积审视

要求：

- 演示页面不能继续留在主入口产物中
- 后台首页首屏资源应明显小于当前演示主页

### 交付物

- 新目录结构
- 类型体系
- 环境配置
- 基础测试
- mock 方案
- 构建优化

### 验收标准

- 新增功能时目录归属清晰
- 不同环境配置切换不需要改代码
- 核心链路有最小自动化保障

---

## P3：模板增强项

### 目标

在基础模板稳定后，增强复用价值和产品完成度。

### 可选项

#### 5.3.1 国际化

适用于需要中英文后台模板输出的场景。

#### 5.3.2 主题模式

可做：

- 明暗主题
- 品牌色扩展
- 侧边栏主题变体

要求：

- 仍然以现有 token 系统为唯一主题源

#### 5.3.3 更完整的通用组件库

可补齐：

- `PageHeader`
- `SearchPanel`
- `StatusTag`
- `PermissionGuard`
- `EmptyBlock`
- `AsyncButton`
- `ConfirmAction`

#### 5.3.4 仪表盘可配置能力

适用于模板展示力增强，但不应前置。

#### 5.3.5 国际化字典、图标注册、权限码生成工具

这些属于增强项，必须在主链路稳定后再做。

---

## 6. 推荐实施顺序

按最短路径推进，建议按下面顺序执行：

1. 先做 P0 的路由、布局、登录、会话、请求层
2. 再做 P1 的菜单、面包屑、标签页、Dashboard、列表页模板、表单页模板
3. 再做 P2 的目录重构、类型、环境、测试、mock、构建优化
4. 最后再做 P3 的主题增强、国际化和高级通用组件

原因：

- P0 决定架构方向
- P1 决定模板是否能承载后台开发
- P2 决定模板是否能长期演进
- P3 只影响上限，不影响模板成立

---

## 7. 里程碑建议

### 里程碑 A

完成 P0 后，项目应达到：

- 可以登录
- 可以进入后台首页
- 可以被路由守卫保护
- 可以发起标准化请求
- 可以承载后台页面开发

### 里程碑 B

完成 P1 后，项目应达到：

- 可以快速新增菜单页面
- 可以快速生成列表页
- 可以快速生成表单页
- 可以作为内部后台项目起始模板

### 里程碑 C

完成 P2 后，项目应达到：

- 可以稳定多人协作
- 可以在不同环境下交付
- 可以依赖基础测试进行回归

---

## 8. 当前最建议立刻开始的工作

如果现在直接开始开发，我建议先做以下 5 件事：

1. 重构路由为 `登录区 + 后台区 + 异常页`
2. 新建 `AdminLayout`，替换当前 `App.tsx` 的演示切换逻辑
3. 重写登录页为后台模板登录页，接入会话管理
4. 标准化请求层，并确定服务端状态与客户端状态边界
5. 选定 `antd` 作为页面主组件体系

这 5 件事完成后，项目就会正式进入“后台模板建设”阶段。

---

## 9. 不建议现在做的事

为避免方向偏移，当前阶段不建议优先做：

- 大量补视觉演示页面
- 提前做复杂 dashboard 配置化
- 提前做国际化
- 提前做一堆业务无关的高级组件
- 同时维护两套页面级 UI 体系

这些都会拉长路径，但不会让模板更早成立。

---

## 10. 后续执行方式建议

建议后续按阶段推进，每一轮只做一个优先级段，不并行摊大饼。

推荐节奏：

- 第一轮：只落 P0
- 第二轮：只落 P1
- 第三轮：只落 P2
- 第四轮：按需要挑选 P3

这样可以确保每一轮结束后，项目都处于可运行、可验证、可继续扩展的状态。
